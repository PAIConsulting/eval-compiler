# Evaluation Compiler — Demo

This is a working demonstration of an automated tool for compiling
Gravity Forms evaluation responses into per-domain Word reports.

## What's in this folder

| File | What it is |
|------|-----------|
| `generate_fake_data.py` | Creates a synthetic CSV of fake responses for testing |
| `fake_responses.csv` | Synthetic data, generated by the script above |
| `keywords.yaml` | Example keyword dictionary for categorizing free-text |
| `compile_report.py` | The main tool — reads CSV + keywords, produces Word reports |
| `reports/` | Output folder containing one Word doc per domain |
| `requirements.txt` | Python packages needed to run this |
| `README.md` | This file |

**ALL DATA IN THIS DEMO IS FICTIONAL.** The fake responses, fake topic
names, fake comments, and fake keyword dictionary are all examples for
demonstration only. None of it resembles real evaluation content.

## Quick start

### One-time setup

You need Python 3 and a few packages. Run this once:

```bash
pip install python-docx pyyaml matplotlib
```

### Running the demo

From this folder:

```bash
# Step 1: Generate the fake CSV (already done; only re-run if you want fresh fake data)
python generate_fake_data.py

# Step 2: Compile the reports
python compile_report.py
```

Output goes into `reports/` — one `.docx` file per domain.

## What the tool does (in plain English)

The tool reads the CSV and, for each domain, produces a Word document with:

1. **Title page** — Domain name, total respondents, percent first-time attendees
2. **Question 2 page** — Horizontal bar chart showing top safety topics by votes
3. **Questions 3, 4, 5 pages** — Each contains:
   - A bar chart showing which subjects respondents picked
   - A breakdown table showing topics within each subject
   - Two-column layout of "Further Comments" categorized by Keyword 1 / Keyword 2
4. **Questions 6, 7, 8 pages** — Open-ended free-text answers, grouped by Keyword 1
5. **Appendix** — Quick stats and a count of comments needing manual review

## How keyword categorization works

The file `keywords.yaml` is the "secret weapon." Each entry says:

> "If a comment contains any of these trigger words, tag it as Keyword 1 = X, Keyword 2 = Y"

Example:

```yaml
- keyword_1: "Personnel"
  keyword_2: "Fatigue"
  triggers: [fatigue, fatigued, tired, exhausted, sleep, rest]
```

When the tool sees a comment like *"We have noticed increased issues related to fatigue"*,
it matches the trigger "fatigue" and tags the comment as **Personnel / Fatigue**.

If multiple categories' triggers match, the one with the most matches wins.
If no triggers match, the comment is tagged **Uncategorized / —** for manual review.

## How to customize

### Update the keywords (most common change)
Edit `keywords.yaml`. Add new categories, add new trigger words, or remove ones that are too aggressive. After 2-3 meetings, this dictionary will become very accurate for your reports.

### Change the visual styling
Edit the constants at the top of `compile_report.py`:
- `COLOR_HEADER`, `COLOR_ACCENT` — header text colors
- `COLOR_LIGHT_BG`, `COLOR_DARKER_BG` — background colors for headers/sections

### Change report structure
The function `build_report()` in `compile_report.py` controls the document layout. Each section of the report is one block of code you can rearrange or modify.

### Use real data (Phase 2)
1. Export entries from Gravity Forms as CSV
2. Map the column names to what `compile_report.py` expects (or rename columns in code)
3. Run the tool
4. Review the output and the "needs manual review" appendix
5. Make any final hand edits in Word

## Limitations of this demo

- **Synthetic data only.** The tool will need adjustment when real CSV columns from Gravity Forms have different names than the demo expects.
- **Generic keyword dictionary.** Real categorization quality depends on a dictionary built from your domain expertise.
- **Generic Word styling.** Branding, fonts, and exact layouts can all be tuned to match your existing report template.
- **No conditional logic awareness.** The tool processes all responses equally; it doesn't know that some respondents wouldn't see certain questions.

These are all addressable in Phase 2 (the real build). This demo is for
seeing the shape of the solution and getting approval to proceed.

## What this demo proves

✅ The architecture works end-to-end: CSV in → categorized Word reports out
✅ Charts and tables generate cleanly from structured data
✅ Free-text categorization with a keyword dictionary is practical
✅ One report per domain, with consistent structure, is achievable
✅ Manual review burden is contained and measurable (appendix flag)
✅ The tool runs entirely on your local machine — no cloud, no AI calls

## What this demo does NOT prove

❌ That the keyword dictionary can be made accurate enough for production use without significant tuning
❌ That all edge cases in real Gravity Forms exports are handled
❌ That the visual output matches your existing report template exactly

These are honest gaps, addressable in Phase 2.

## Audit notes

This codebase is designed to pass standard security and supply-chain audits
(CodeQL, SBOM tools, etc.) cleanly. Specifically:

- **No AI/ML libraries.** No imports of openai, anthropic, transformers,
  torch, tensorflow, langchain, or any AI/ML library.
- **No network calls.** No imports of requests, urllib, socket, httpx, or
  any networking library. The tool runs entirely offline.
- **No shell execution.** No use of subprocess, os.system, or shell=True.
- **No dynamic code execution.** No use of eval(), exec(), or compile().
- **Safe YAML parsing.** Uses yaml.safe_load() instead of yaml.load() to
  prevent arbitrary code execution from malicious YAML.
- **Pinned dependencies.** All dependency versions pinned in
  requirements.txt for deterministic, auditable builds.
- **Open-source dependencies only.** All third-party libraries are widely
  used, well-maintained, and have permissive licenses (MIT or PSF-like).
- **Explicit license.** See LICENSE file at the project root.
